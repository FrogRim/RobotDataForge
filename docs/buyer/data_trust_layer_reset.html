왜 바꿨나

기존 방향은 민간 작업자가 쉽게 로봇 데이터를 만들 수 있다는 장기 목표와 맞닿아 있었지만, 1차 제품 증명 경로가 Quest handtracking, ALVR, SteamVR/OpenXR, Isaac XR UI에 지나치게 묶였다. 구매자가 사는 것은 HMD 자체가 아니라, 신뢰 가능한 데이터와 그 증거다.

MVP-1 lesson

Task success와 training eligibility를 분리했다

사람이 작업을 끝내도 action semantics, replay 가능성, quality gate, curation evidence, export/trainer compatibility가 없으면 구매 가능한 데이터가 아니다.

Gate 0 evidence

입력 스트림 안정성이 task success보다 앞선다

Gate 0은 task success를 무시하고 hand tracking rate, XR frame validity, raw wrist jump, tracking loss, recenter instability 같은 입력 truth를 먼저 검증한다.

Reset decision

첫 proof를 HMD-free로 옮겼다

HMD/OpenXR 코드는 삭제하지 않고 experimental input adapter로 격하시켰다. 대신 데이터 신뢰 레이어 자체가 작동하는지 scripted/synthetic replay fixture로 검증한다.

무엇이 바뀌었나

이번 변경의 핵심은 기능 추가보다 claim boundary 재정렬이다. HMD 경로를 버린 것이 아니라, 제품의 생사를 좌우하는 기본 proof path에서 분리했다.

바꾸지 않은 것

리셋은 방향 전환이지 삭제가 아니다. 장기적으로 일반 작업자가 로봇 데이터를 만들어 판매할 수 있게 한다는 목표는 유지하되, 첫 구매자 신뢰 증명은 HMD 안정성에 의존하지 않는다.

Claim boundary

Gate A physical collection은 아직 재개하지 않는다. Quest/OpenXR/HMD 경로는 보존하지만 experimental adapter이며, physical HMD Gate 0 rerun은 implementation acceptance가 아니라 별도의 release validation이다. MVP-1은 policy improvement를 주장하지 않고, MVP-2에서 downstream policy performance를 검증한다.

앞으로의 공개 내러티브

LinkedIn의 다음 글은 Quest handtracking 실패담에 머무르기보다, 실패를 통해 제품 정의가 선명해졌다는 쪽으로 가져가는 것이 맞다. 핵심 문장은 단순하다.

ForgeXR / RDF is not an HMD demo. It is a robot data trust layer. Bad input is blocked, rejected data is explained, accepted data is replay-verified, and every buyer-facing artifact carries provenance, schema, limitations, curation, export, and trainer-loader evidence.
