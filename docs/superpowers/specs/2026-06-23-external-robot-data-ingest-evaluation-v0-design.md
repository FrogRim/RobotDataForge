# Spec: External Robot Data Ingest / Evaluation v0

Date: 2026-06-23
Status: RALPLAN CONSENSUS APPROVED; READY FOR ULTRAGOAL
Branch: TBD

## Objective

이번 관문의 목적은 Robot Data Forge가 내부 proof fixture나 controlled Isaac package만
평가하는 시스템이 아니라, 외부에서 제공되었거나 독립적으로 기록된 robot-action log를
받아 같은 trust layer에 태울 수 있는지 증명하는 것이다.

이번 slice는 robot control, live runtime integration, policy learning uplift를 열지 않는다.
증명하려는 것은 다음 하나다.

```text
Given one externally supplied recorded robot command/state log with clear
provenance, Robot Data Forge can ingest the source files, preserve source hashes,
emit normalized trajectory contracts, evaluate/curate accepted and rejected data,
export trainer-loadable HDF5 when eligible, run trainer smoke, and ship a
self-contained verifier package whose pass/fail verdict can be recomputed from
included evidence without trusting the prover's local storage.
```

이 관문은 MVP-2의 `learning-proven` claim이 아니다. MVP-3B/C의
source/embodiment infrastructure proof 다음에 오는 practical data evaluation proof다.

## Current Repo Facts

- MVP-2, MVP-3A, MVP-3B, MVP-3C proof packages are tracked under `docs/proof/`
  and have stdlib verifier scripts.
- `storage/` artifacts are not an external source of truth.
- The only tracked source-like non-proof input is the MVP-1+ UR JSONL fixture:
  `fixtures/mvp1plus/universal_robots_ur_recorded_log_fixture/`.
- That fixture explicitly says it is repo-local/generated fixture evidence, not
  actual external robot data.
- `RobotEmbodimentAdapterRegistry.project_source_evidence()` already accepts a
  `metadata.json` + `accepted_command_state.jsonl` + `rejected_command_state.jsonl`
  source directory and projects it into RDF trajectories/evaluations/contracts.
- `scripts/run_mvp1plus_embodiment_proof.py` already supports `--ur-recorded-log-dir`,
  which is the lowest-risk existing hook for a user-supplied recorded log directory.
- `scripts/export_rdf_to_hdf5.py` and `scripts/run_mvp1_trainer_smoke.py` provide
  the existing offline export + loader-smoke path.

## Brainstorming

### Option A: External JSONL Command-State Drop

```text
source_kind=external_jsonl_command_state_drop
input=metadata.json + accepted_command_state.jsonl + rejected_command_state.jsonl
first_target=UR-style command/state log exported offline from an external recording
```

Optimizes:

- Uses the existing RDF source-evidence shape and adapter hook.
- Avoids live UR/RTDE, ROS2/DDS, Isaac, HMD/OpenXR, and heavy public dataset
  dependencies.
- Makes auditability straightforward: the verdict-critical source rows are small
  enough to include in `docs/proof/.../data/`.
- Lets RDF prove the data trust layer before proving every upstream parser.

Tradeoff:

- Requires the external party or user to provide/export logs into RDF's JSONL
  command/state drop format.
- Does not prove native UR RTDE, ROS bag, Franka, or LeRobot parser coverage.

### Option B: LeRobot v3 Public Dataset Slice

```text
source_kind=public_lerobot_dataset_slice
input=local LeRobot v3 snapshot or downloaded public dataset slice
```

Optimizes:

- LeRobot v3 is a standardized robotics dataset format with tabular state/action
  data, timestamps, video, and metadata.
- Public datasets make provenance easier to share externally.
- Strong portfolio story: RDF can evaluate an ecosystem-standard dataset source.

Tradeoff:

- Requires Parquet/metadata reading and likely new dependencies (`pyarrow`,
  `datasets`, or `lerobot`).
- Many public datasets include large video shards; v0 would need a carefully
  bounded state/action-only audited slice.
- Licensing and redistribution terms must be checked before committing source
  slices into `docs/proof/`.

### Option C: DROID / BridgeData Slice

```text
source_kind=public_large_robot_dataset_slice
input=DROID, BridgeData, or converted public dataset sample
```

Optimizes:

- Strongest "real-world robot data" credibility.
- DROID and BridgeData are well-known external robot learning datasets.

Tradeoff:

- DROID and BridgeData are large and conversion-heavy.
- Download, license, redistribution, action-space mapping, and storage concerns
  are bigger than the v0 proof needs.
- It risks turning the slice into dataset archaeology instead of RDF trust-layer
  proof.

### Option D: Live Runtime Adapter

```text
source_kind=live_rtde_or_ros2_or_franka
```

Optimizes:

- Strongest integration signal if successful.

Tradeoff:

- Requires live robot, ROS2/DDS, RTDE, or hardware access.
- Violates current stop conditions.
- Blurs data evaluation with robot control/readiness.

## Decision

External Robot Data Ingest / Evaluation v0 uses **Option A: External JSONL
Command-State Drop**.

The first concrete source should be a user-supplied or independently recorded
UR-style command/state log exported into RDF's JSONL drop format. It must not be
generated by RDF proof scripts, copied from `docs/proof/`, or mislabeled from
the existing fixture.

If such a source is not available during implementation, the slice may only close:

```text
external_ingest_contract_ready
```

It must not close:

```text
external_data_evaluated
```

LeRobot v3 is the preferred next public-dataset adapter after v0 because its
format is explicitly state/action/timestamp/metadata based. DROID or BridgeData
can become later audited-slice adapters after dependency, storage, and license
decisions.

## Claim Tiers

### Tier 0: Contract Ready

```text
package_status=external_ingest_contract_ready
```

Allowed when:

- importer/verifier/schema/tests exist;
- synthetic or repo-local fixture tests pass;
- no externally supplied or public dataset source is included.

Not allowed to claim:

- external data evaluated;
- real-world robot data evaluated;
- public dataset compatibility;
- live runtime support.

### Tier 1: External Data Evaluated

```text
package_status=external_data_evaluated
```

Allowed only when:

- source metadata declares `source_origin` as `external_supplied_recorded_log` or
  `public_dataset_recorded_log`;
- source metadata declares `provenance_trust_tier`;
- `attested_file_drop` includes a named responsible source owner and license
  statement;
- `refetchable_public_source` includes `public_source_url`,
  `upstream_dataset_revision`, and `upstream_published_sha256`;
- source files are not under `docs/proof/`, `storage/`, generated output dirs, or
  known repo fixture dirs at ingest time;
- provenance/license fields are present and not placeholder values;
- verdict-critical source rows are copied into `docs/proof/.../data/source/`;
- the verifier recomputes pass/fail from included source rows and projected
  artifacts;
- buyer report and README clearly state that this is data evaluation, not robot
  readiness.

Temp external-like sources used by tests are not closure evidence. They may
exercise verifier mechanics in temporary packages, but the canonical
`docs/proof/...` package may claim `external_data_evaluated` only with actual
external/public source rows or a deterministic audited slice.

### Tier 2: Native Parser Adapter

```text
package_status=native_external_parser_evaluated
```

Out of scope for v0. This tier would prove a native parser for LeRobot, DROID,
ROS bag, UR RTDE CSV, Franka logs, or another upstream format.

## Provenance Trust Boundary

This slice has a deliberate provenance boundary.

`external_data_evaluated` means:

```text
RDF ingested a recorded log that is attested as external, preserved its source
hashes, normalized it into RDF contracts, curated it, exported eligible data,
ran trainer smoke, and shipped a self-contained package whose data-evaluation
verdict is independently recomputable from included evidence.
```

It does not mean:

```text
The offline verifier cryptographically proved that the rows physically came from
a real external robot, partner lab, public dataset, UR controller, Franka
controller, ROS bag, RTDE stream, or any other upstream event.
```

The verifier can recompute what follows from the dropped evidence:

```text
source JSONL -> counts/quality -> projection -> contract -> curation -> HDF5
-> trainer smoke -> package verdict
```

The verifier cannot prove the external origin of self-asserted metadata. A
malicious actor could hand-write JSONL rows and claim `source_owner="ACME
Robotics"` with `generated_by_rdf=false` and `repo_fixture=false`. The verifier
must reject obvious fixture/generated misuse and incomplete provenance, but it
cannot eliminate sophisticated fabrication without an independently verifiable
upstream binding.

Therefore v0 has two provenance trust tiers:

```text
trust_tier=attested_file_drop
  The source owner/license/provenance statement is present and hash-bound, but
  external origin remains an out-of-band human/legal attestation.

trust_tier=refetchable_public_source
  The package records a public source URL plus upstream revision/content hash, so
  an auditor can independently fetch the source and compare hashes.
```

For `attested_file_drop`, the package may claim only that RDF evaluated a
source that was provided and attested as external. It must not phrase this as
"RDF proved the data came from a real robot."

For `refetchable_public_source`, the verifier should check the declared URL,
dataset revision, and upstream hash fields for presence and consistency. Actual
network refetch can be a reviewer procedure or optional verifier mode, but the
hash-bound public coordinates must be present in the package.

This boundary is part of the product claim. Buyer-facing reports must state it
explicitly.

## Source Eligibility Contract

The v0 source directory must have this shape:

```text
external_source_dir/
  metadata.json
  accepted_command_state.jsonl
  rejected_command_state.jsonl
  PROVENANCE.md        # recommended
  LICENSE.txt          # required for public datasets, recommended for partner logs
```

`metadata.json` is raw external source metadata. It must be validated as source
attestation evidence and must not be mutated to satisfy RDF adapter internals.
It intentionally does not need current `project_source_evidence()` adapter-only
fields such as `adapter_version`, `evidence_level`, `source_provenance`, or
`limitations`.

`metadata.json` required fields:

```json
{
  "schema_version": "rdf_external_command_state_source_metadata_v0.1.0",
  "source_id": "external_ur_log_YYYYMMDD_slug",
  "source_origin": "external_supplied_recorded_log",
  "source_acquisition": "file_drop",
  "source_owner": "person_or_org_or_public_dataset",
  "source_license": "private_review|public_license_name|not_redistributable",
  "source_redistribution_allowed": true,
  "provenance_trust_tier": "attested_file_drop|refetchable_public_source",
  "public_source_url": null,
  "upstream_dataset_revision": null,
  "upstream_published_sha256": null,
  "recorded_log_backed": true,
  "generated_by_rdf": false,
  "repo_fixture": false,
  "robot_family_claimed": "universal_robots_ur",
  "embodiment_class_claimed": "industrial_arm",
  "command_stream": {
    "interface": "industrial_arm_command_state",
    "unit": "normalized_delta_pose"
  },
  "state_stream": {
    "interface": "industrial_arm_state_stream"
  },
  "coordinate_frames_declared": {
    "command_frame": "task_frame",
    "state_frame": "robot_base_frame"
  }
}
```

Each JSONL row must be a JSON object with:

```text
sequence_id
timestamp
task_phase
command.interface
command.unit
command.vector
state.interface
state.joint_positions
state.end_effector_position
state.end_effector_quaternion
action_semantics.representation
action_semantics.coordinate_frame
action_semantics.normalized_contract_roles
quality.action_contract_valid
quality.replay_verified
quality.control_quality
quality.rejection_reason
```

Accepted rows require:

```text
quality.action_contract_valid=true
quality.replay_verified=true
quality.control_quality=pass
quality.rejection_reason=null
```

Rejected rows require at least one buyer-readable rejection reason:

```text
quality.action_contract_valid=false OR
quality.replay_verified=false OR
quality.control_quality=fail OR
quality.rejection_reason != null
```

For v0, the included source evidence or deterministic audited slice must contain:

```text
accepted_rows >= 4
rejected_rows == 1
```

This is an explicit v0 constraint, not a product-level limitation. The existing
`project_source_evidence()` path projects all accepted rows but one rejected row,
so v0 uses exactly one included rejected row to preserve current adapter semantics
without redesigning the projection contract. If an external source has multiple
rejected rows, the v0 package must choose one rejected row by a deterministic
audited-slice rule and disclose that the verdict applies to the included slice.
If a source has no rejected rows, v0 cannot close the full accepted/rejected
external evaluation package; it may only close `external_ingest_contract_ready`
or stop for a future accepted-only evaluation slice.

## Metadata Staging / Enrichment Boundary

The raw source metadata and the adapter-compatible metadata are different
artifacts.

Existing projection code expects adapter metadata in the source directory:

```text
metadata.adapter_id
metadata.adapter_version
metadata.robot_family
metadata.embodiment_class
metadata.command_state_interface
metadata.state_interface
metadata.coordinate_frames
metadata.evidence_level
metadata.source_provenance
metadata.claim_boundary
metadata.limitations
```

External v0 must not require a partner/public source to author those RDF
adapter-internal fields directly. Instead, the runner must build deterministic
staging input:

```text
raw external metadata + selected adapter profile constants
  -> adapter-compatible staging metadata
  -> project_source_evidence(staging_source_dir)
```

Required package artifacts:

```text
data/source/metadata.json                  # immutable raw external metadata
data/source/accepted_command_state.jsonl   # immutable included raw rows/slice
data/source/rejected_command_state.jsonl   # immutable included raw rows/slice
data/staging/metadata.json                 # derived adapter-compatible metadata
data/staging/accepted_command_state.jsonl  # byte-identical copy of source accepted rows
data/staging/rejected_command_state.jsonl  # byte-identical copy of source rejected rows
data/staging/staging_derivation_report.json
```

The staging derivation report must record:

```text
raw_metadata_sha256
source_accepted_sha256
source_rejected_sha256
selected_adapter_id
selected_adapter_profile_version
derivation_algorithm
derived_metadata_sha256
staging_accepted_sha256
staging_rejected_sha256
field_mapping
adapter_profile_constants
```

The verifier must treat `data/source/metadata.json` and source JSONL rows as the
source of truth, recompute the staging metadata derivation, and assert:

```text
sha256(data/staging/accepted_command_state.jsonl)
  == sha256(data/source/accepted_command_state.jsonl)

sha256(data/staging/rejected_command_state.jsonl)
  == sha256(data/source/rejected_command_state.jsonl)

sha256(recomputed_adapter_metadata)
  == sha256(data/staging/metadata.json)
```

Tampering either raw metadata or staging metadata must fail. Staging metadata
without raw source metadata must fail. This boundary prevents RDF from silently
rewriting external provenance while still allowing the current adapter projection
contract to be reused.

## Pipeline Contract

The implementation path must remain offline:

```text
external_source_dir
  -> source adapter / import adapter
  -> deterministic adapter-compatible staging input
  -> source file hash manifest
  -> NormalizedTrajectoryContract
  -> NormalizedTrajectoryContractValidator
  -> evaluator consistency checks
  -> curation manifest with accepted/rejected reasons
  -> HDF5 export for eligible accepted trajectories
  -> trainer smoke on exported HDF5
  -> buyer-readable report
  -> self-contained proof package
  -> stdlib verifier
```

No live robot control, no live ROS2/DDS, no HMD/OpenXR, no Isaac rerun, and no
policy training are part of v0.

The source rows included under `data/source/` are the source of truth for the
verdict. Staging artifacts are deterministic adapter inputs derived from that
source evidence. Summaries, buyer reports, and package manifests are
cached/index artifacts only.

When the package only claims:

```text
package_status=external_ingest_contract_ready
```

it must set:

```text
external_source_included=false
```

and must not include canonical external source rows. It may include schemas,
templates, and test fixture mechanics outside the canonical proof package, but
those cannot close `external_data_evaluated`.

If the full external source is too large or cannot be redistributed, v0 must
define an audited slice:

```text
audited_slice_selection_rule=first_n|deterministic_stride|sha256_ordered_prefix
audited_slice_selection_parameters=hash_bound_parameters
audited_slice_rows=included_in_docs_proof_data
full_source_hash=recorded
full_source_verdict_claimed=false
audited_slice_verdict_claimed=true
```

The verifier must enforce that the audited slice declares a deterministic rule
and that the included slice is internally consistent with that rule. If the full
source is not included, the package must disclose that cherry-picking cannot be
fully eliminated by the offline verifier and that the verdict applies only to
the included audited slice.

## Verifier Contract

Create a dedicated verifier:

```text
scripts/verify_external_robot_data_ingest_package.py
```

The verifier must be stdlib-only.

It must recompute from included evidence:

- package file hashes;
- source file hashes;
- source eligibility and provenance fields;
- raw metadata to staging metadata derivation;
- staging row bytes are identical to source row bytes;
- provenance trust tier fields and minimum owner/license/public binding
  requirements;
- source rows are parseable JSONL objects;
- accepted/rejected row counts;
- timestamp monotonicity per source file;
- command/state vector numeric shape;
- action semantics roles include `learning_action` only as a data label, not a
  learning-proven claim;
- accepted row quality predicates;
- rejected row rejection predicates and buyer-readable reasons;
- source rows projected into trajectory frame counts without row loss;
- normalized contract validator report agrees with source/projection evidence;
- curation manifest accepted/rejected counts and reasons;
- HDF5 exists in the package when the package claims trainer-loadable export;
- HDF5 inspection/trainer smoke reports agree with included HDF5;
- non-claim keys are present and false;
- README/report/config/package JSON do not leak forbidden claims;
- no simulation held-out closure seed range is opened;
- spent ranges `40000-40049` and `42000-42049` are verifier-owned constants,
  not package-config declarations; any seed-like field in the package must be
  disjoint from those fixed ranges.

The verifier must fail if:

- source origin is fixture/generated but package claims external data evaluated;
- `generated_by_rdf=true`;
- `repo_fixture=true`;
- provenance owner/license fields are placeholder strings;
- `refetchable_public_source` omits public URL, upstream revision, or published
  hash;
- source rows are absent and only hashes are present;
- staging metadata is present without raw source metadata;
- staging metadata derivation does not match raw metadata and adapter constants;
- staging rows differ from source rows;
- audited slice claims full-source verdict when full source rows are not
  included;
- source hash is refreshed after source tampering but projection/contract data no
  longer matches source rows;
- accepted/rejected labels are flipped without matching quality predicates;
- buyer report claims real robot success, physical readiness, live runtime
  support, policy uplift, deployment, marketplace, production, or sim-to-real.

## Stop Conditions

Stop and report instead of closing `external_data_evaluated` if:

- no external source is available. This does not block
  `external_ingest_contract_ready`;
- source provenance, ownership, or license is unclear;
- source cannot be redistributed and no audited slice can be included;
- source lacks action and state semantics;
- included source evidence does not satisfy the v0 row-count contract
  `accepted_rows >= 4` and `rejected_rows == 1`;
- source rows cannot be mapped without weakening `NormalizedTrajectoryContractValidator`;
- HDF5 export would require dropping embodiment/action semantics;
- package would contain only hashes to ignored local artifacts;
- fixture/generated evidence would be promoted as external data;
- live robot, live ROS2/DDS, HMD/OpenXR, Isaac rerun, or long policy training is
  required before the offline contract is clear;
- buyer-facing report would imply real robot readiness or policy uplift.

```text

C4 HDF5 export + trainer smoke integration
```

## Testing Strategy

Tests must be TDD-first.

Add tests for:

- external source validator rejects repo fixtures and generated RDF outputs when
  package status claims `external_data_evaluated`;
- validator accepts a temp external JSONL source outside fixture/proof/storage
  paths with complete provenance as a test double only;
- source file hashes are recorded;
- accepted JSONL rows and exactly one included rejected JSONL row are parsed and
  counted;
- accepted row quality predicates are enforced;
- rejected row reasons are preserved;
- projection emits normalized trajectory contract;
- `NormalizedTrajectoryContractValidator` passes/fails correctly;
- HDF5 export and trainer smoke pass for eligible accepted data;

- tampering with source rows, labels, hashes, projection, contract, non-claims,

## Success Criteria

The slice is successful only if one of these two outcomes is honestly produced.

### Outcome A: Contract Ready

```text
package_status=external_ingest_contract_ready
```

Allowed when source adapter/verifier tests pass, but no actual external source
is available. This is useful but not a final external-data proof.

### Outcome B: External Data Evaluated

```text
package_status=external_data_evaluated
```

Allowed only when an actual external/public recorded source or deterministic
audited slice is included in `docs/proof/.../data/`, and the verifier can
recompute the verdict from included source rows.

## Open Questions for RALPLAN

1. Will the first real source be a user-supplied UR-style JSONL file drop, or a
   public dataset slice converted into RDF JSONL?
2. If user-supplied, what redistribution level is allowed: commit full source
   rows, commit audited slice only, or hash-lock original out-of-band?
3. Should the first implementation create only the generic JSONL importer, or
   also a LeRobot v3 parser behind an optional dependency decision?
4. Should v0 keep the conservative row-count contract
   `accepted_rows >= 4`, `rejected_rows == 1`, or should a later slice enhance
   `project_source_evidence()` to preserve arbitrary rejected-row counts?
5. Should this milestone be named `External Robot Data Ingest v0`, `MVP-4A`, or
   a post-MVP-3 proof addendum?
