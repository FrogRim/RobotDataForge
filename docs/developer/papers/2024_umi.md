# Universal Manipulation Interface

## Overview

- problem: RDF needs to convert raw human teleoperation into policy-ready data while reducing coordinate/calibration brittleness.
- why it matters: UMI emphasizes relative trajectory action representation, latency matching, replay buffer generation, and dataset robustness. These are directly useful for RDF export and training.

## System Architecture

- components:
  - handheld gripper with camera.
  - SLAM pipeline.
  - replay buffer generation.
  - Diffusion Policy training.
  - deployment/evaluation scripts.
- data flow:
  - human demonstration is collected with a portable interface.
  - SLAM reconstructs camera/gripper trajectory.
  - `dataset.zarr.zip` replay buffer is generated.
  - policy is trained on the replay buffer.

## Data Pipeline (MOST IMPORTANT)

- input (VR signal):
  - UMI does not use Quest VR as primary input, but it solves the same human demonstration -> policy data problem.
- processing (mapping -> robot action):
  - use relative trajectory action representation.
  - match inference-time latency to collection-time latency.
- output (dataset):
  - Zarr replay buffer.
  - image + gripper pose/action data.

## Dataset Structure

- state:
  - visual observation.
  - gripper/pose state.
- action:
  - relative trajectory actions rather than fragile global absolute poses.
- observation:
  - camera frames.
- timestamp:
  - SLAM/camera/trajectory synchronization.

## Logging Strategy

- frequency:
  - camera/pose stream.
  - success of SLAM is tracked as a dataset quality measure.
- storage format:
  - Zarr replay buffer for training.
  - RDF can initially export JSON but should add Zarr/LeRobot conversion path.

## Implementation Details

- UMI has a post-processing pipeline ending in `07_generate_replay_buffer.py`.
- It explicitly reports low SLAM success as a quality issue and guides users to fix collection procedure.
- It trains Diffusion Policy directly from processed replay buffer.

## Minimal Integration Plan

1. In `build_frame()`, compute `action.delta_position` from previous saved frame.
2. Store `relative_end_effector_delta` and `relative_object_delta`.
3. In export payload, include both `absolute_action` and `relative_action`.
4. Add a training-prep script that normalizes actions and writes `dataset_stats.json`.

## Expected Impact

- Reduces brittleness from viewpoint/anchor drift.
- Makes RDF data easier to use with Diffusion Policy/ACT-like models.
- Creates clear path from raw trajectory to training replay buffer.

## What to Adopt

- relative action representation.
- latency metadata as a training concern.
- replay buffer generation as a separate post-processing step.

## What to Avoid

- copying UMI hardware/SLAM pipeline.
- depending on SLAM for MVP-0 since Isaac state is directly available.
