Implementation Update: Offline Provenance Analyzer

같은 날짜에 PR 1 범위의 오프라인 analyzer가 구현되었다. scripts/analyze_hmd_motion_mapping.py는 이제 provenance_timeline을 출력하며, raw wrist, aligned wrist, raw wrist origin, wrist offset, desired target, executed command, actual EEF, object, peg, hole, hole target 단계 중 어느 단계에서 discontinuity가 처음 나타나는지 기록한다.

저장된 traj_7f78c4bbd77e 분석 결과는 transition_count=179, event_count=23, first_stage_counts={'raw_wrist': 22, 'actual_eef': 1}이다. 이 결과는 저장된 trajectory에서는 대부분의 discontinuity가 raw wrist 단계에서 먼저 관측된다는 뜻이다. 다만 물리적 원천은 여전히 HMD 검증 전까지 확정할 수 없다.

검증: synthetic HMD-free tests 6 passed, 전체 diagnostic script tests 60 passed, 6 skipped, ruff check와 py_compile 통과.

2. HMD 테스트 미수행

실제 HMD 테스트나 신규 수집은 수행하지 않았다. run_hmd_axis_debug.sh gate0-all, collection loop, Quest/OpenXR 런타임이 필요한 명령도 실행하지 않았다.

3. 현재 Gate 중심 구조

  • scripts/rdf_input_sources.py: source-agnostic WristPoseSample와 Quest/OpenXR adapter.
  • scripts/run_gate0_xr_input_viability.py: tracked rate, XR-valid rate, raw wrist jump, tracking loss, latency/frame drop 등을 평가.
  • scripts/run_collection_loop.sh: fresh gate0-all 통과 없이는 Gate A collection 차단.
  • apps/api/app/services/evaluator.py: task failure와 data/input/control quality failure를 분리.
  • apps/api/app/services/curator.py: evaluator reject reason을 curation manifest에 보존.

4. Gate가 풀지 못하는 문제

  • raw wrist jump의 발생 원인이 Quest handtracking, ALVR/SteamVR, Isaac OpenXR cache, recenter/rebase, simulator boundary 중 어디인지 설명하지 못한다.
  • valid=true가 timestamp freshness, confidence, velocity plausibility, control safety를 의미하지 않는다.
  • held frame의 zero action이 learning label로 어떤 의미인지 frame 단위로 충분히 분리하지 못한다.
  • 구버전 artifact와 최신 schema 사이의 provenance drift를 해결하지 못한다.

5. Raw 입력에서 Action label까지의 실제 코드 경로

  1. teleop_se3_agent.py::teleop_right_wrist_pose_list()가 teleop_interface._previous_joint_poses_right["wrist"]를 읽는다.
  2. rdf_pose_is_valid()가 finite/nonzero/non-anchor-fallback만 검사한다.
  3. RdfRawWristDirectEeTargetController.apply()가 raw_offset = current_raw - raw_wrist_origin을 계산한다.
  4. rdf_map_position_delta_xyz()가 axis/yaw mapping을 적용한다.
  5. RdfOperatorFollowController.apply()가 bounded direct-EE target과 native Isaac action을 만든다.
  6. RdfIsaacRuntimeRecorder._action_payload()가 teleop_intent, executed_control, learning_action, retargeted_robot_action을 분리 저장한다.
  7. evaluator.py와 curator.py가 training eligibility를 결정한다.

7. 현재 자료로 확인 불가능한 항목

  • 현재 Isaac OpenXR interface가 실제 timestamp, confidence, location flags, measured/predicted state를 제공하는지.
  • raw wrist jump가 물리 센서 원본에서 발생했는지, transform/recenter 이후 발생했는지.
  • 최신 live collection에서 모든 frame에 action_hold, hold_reason, tracking_epoch_id가 실제로 저장되는지.
  • H15 scene discontinuity가 IsaacLab reset boundary인지, task-state teleport인지, recorder boundary bug인지.

8. Root cause 후보

  1. 주 후보: input signal modeling 문제. valid와 control_safe가 분리되어 있지 않고 timestamp/confidence provenance가 부족하다.
  2. 보조 후보: recenter/reacquire provenance boundary 문제. rebase와 warmup이 있지만 구버전 artifact에서는 hold/epoch metadata가 일관되게 보존되지 않았다.
  3. 보조 후보: action representation 문제. held zero action이 frame-level learning eligibility와 명확히 묶여 있지 않다.
  4. 별도 후보: simulator/task-state boundary 문제. H15 frame 172에서 task target과 동적 상태가 함께 점프했다.

9. Gate 추가 없이 풀어야 하는 구조적 문제 정의

ForgeXR은 "이 stream이 Gate 0을 통과했는가"만 묻지 말고, raw 측정값, tracking state, timestamp/age/confidence, raw continuity, transformed continuity, recenter epoch, control-safe state, action-label eligibility를 하나의 신호 모델로 분리해 기록해야 한다.

10. 필요한 후보 코드 변경

  • WristPoseSample 또는 별도 InputSignalState에 sample_valid, tracking_valid, control_safe를 분리한다.
  • raw-wrist controller가 teleop 내부 cache를 직접 읽지 않고 normalized sample/state를 소비하도록 정렬한다.
  • OpenXR backend에서 timestamp/confidence를 못 주면 not_available provenance를 명시 저장한다.
  • recenter/reacquire event를 before/after origin, anchor, reason, epoch와 함께 frame timeline에 기록한다.
  • offline provenance analyzer를 추가해 raw, aligned, origin, target, command, EEF jump를 같은 timeline에서 분해한다.
  • held frame의 learning_action을 frame/action-contract level에서 non-learning-eligible로 표시한다.

11. 사용자가 깨어난 뒤에만 가능한 검증

  • provenance instrumentation 이후 static/slow/recenter/reacquire HMD 검증.
  • jump frame에서 raw pose, aligned pose, target, command, actual EEF를 동시 비교.
  • tracking loss 이후 첫 valid frame이 물리적으로 plausible한지 확인.
  • OpenXR/SteamVR가 더 풍부한 tracking status를 줄 수 있는지 확인.
