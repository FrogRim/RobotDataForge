2026-05-17 수정: 직전 run에서는 widget enabled 로그가 있었지만 HMD 안에서는 보이지 않았다. 원인은 XR session 시작 전에 /_xr/stage/xrCamera source prim 없이 widget을 만들어 시야 밖에 배치될 수 있었던 것이다. 이제 widget은 source prim이 실제로 생긴 뒤 lazy-create되며, 생성 성공 시 사용한 source/translation/target을 로그로 남긴다.
Primary HMD UI

XR text widget에서 보여야 할 내용

  • 1RECENTER: MOVE TO START BOX이면 robot EEF/fingertip을 AR start box 안으로 이동.
  • 2RECENTER: HOLD START n/N이면 box 안에서 움직이지 않고 hold.
  • 3RECENTER: OK 또는 green start box면 자동 recenter 완료.
  • 4PHASE: APPROACH/ALIGN/INSERT로 현재 task phase를 확인.
  • 5SUCCESS READY 0.0/1.0s와 progress bar로 practice success hold를 확인.
  • 6AUTO FINALIZE READY가 보이면 keyboard 없이 success finalize가 진행된다.
Not Primary UX

기본 수집에서 쓰지 않을 것

  • -HMD를 쓴 operator에게 terminal 로그를 보라고 하지 않는다.
  • -P/N/R/F hotkey는 debug fallback일 뿐이다.
  • -sphere/cube primitive 상태등은 proper UI가 아니므로 기본 off다.
  • -HMD 안에서 성공 판정이 안 보이면 accepted 수집으로 넘어가지 않는다.
  • -XR text widget이 안 보이면 먼저 UI 문제를 고치고 task data 수집을 미룬다.

통과한 HMD minimum 기준

  • 1HMD에서 robot fingertip이 손 움직임에 납득 가능한 속도로 반응했다.
  • 2손을 약 10cm 움직였을 때 EEF가 같은 방향으로 눈에 띄게 움직였다.
  • 3최신 operator-success recording validator는 passed=true, frame_count 159, issue 0이다.
  • 4calibration analyzer는 issue 0, warning 0, hand tracking rate 1.0을 반환했다.
  • 5desired/applied EE action, native Isaac action, action/replay contract가 저장됐다.

아직 실패한 기준

  • Alatest episode는 operator_success로 finalize됐다.
  • Bevaluator failure reason은 RETARGETING_JUMP이며, 이는 task failure가 아니라 data quality failure로 분리한다.
  • Craw/human_success evidence로는 남지만 curated accepted count는 0이다.
  • Daccepted 승격에는 task success와 action_replay_gate.passed=true가 필요하다.
  • EMVP-1C는 positive held-out policy_success_rate uplift 전까지 닫지 않는다.

Evaluation Semantics Split

이제 `evaluation.success=false`를 곧바로 “사람이 task를 실패했다”로 해석하지 않는다. Raw/operator evidence와 training/proof eligibility를 분리한다.

Task Outcome

작업 결과

  • 1operator_success는 N/finalize 기준이다.
  • 2evaluator_task_success는 true / false / unknown이다.
  • 3object state만으로 안정 판정이 안 되면 false가 아니라 unknown이다.
Data Quality

데이터 품질

  • 1RETARGETING_JUMP는 DATA_QUALITY_FAILURE다.
  • 2sync, replay, action contract, native action saturation을 별도 gate로 본다.
  • 3품질 실패는 raw 저장을 막지 않지만 학습 승격을 막는다.
Curation

승격 기준

  • 1raw_saved, human_success_pool은 관대하게 기록한다.
  • 2training_eligible, curated_accepted, proof_eligible은 엄격하다.
  • 3accepted 승격에는 replay/action/data-quality gate가 필요하다.
Field Latest Value 해석
operator_success true 사용자가 N으로 성공 lifecycle을 남겼다. Raw/human evidence로 보존한다.
evaluator_task_success unknown data quality gate가 먼저 걸렸으므로 task 실패로 단정하지 않는다.
failure_category DATA_QUALITY_FAILURE RETARGETING_JUMP는 task outcome failure가 아니다.
native_action_saturation fail native Isaac action이 자주 포화된다. 이 상태는 accepted training material로 쓰면 안 된다.
curated_accepted false replay/action/data-quality gate 전부 통과해야 true가 된다.

최소 통과선과 operator-success raw evidence는 이번 run에서 충족했다. MVP-1 통과선은 operator가 HMD에서 보고 Peg-in-Hole을 실제로 성공시키고, 그 trajectory가 replay/curation/export/policy A/B proof까지 이어지는 것이다.
