Pre-HMD Step 1 · 돌다리 검증

HMD 실증 전에 입력 신호 안전문을 한 번 더 세웠다

이번 step은 로봇을 더 세게 움직이게 만드는 작업이 아니다. 손목 위치가 튀는 순간을 수집 전에 늦추고, 저장된 뒤에는 학습 후보에서 탈락 이유를 남기는 방어선 작업이다.

0.03 m첫 recenter 전 right-wrist 안정 기준
0.10 mevaluator raw-wrist jump 기본 red flag
0 HMD runs실제 HMD 테스트는 아직 실행하지 않음

Middle-school explanation

아주 쉽게 말하면

HMD 손추적은 로봇에게 “내 손이 여기 있어!”라고 계속 말해준다. 그런데 그 위치가 갑자기 30cm, 90cm씩 순간이동하면 로봇은 따라가면 안 된다. 이번 작업은 그 순간이동을 “진짜 손동작”이 아니라 “입력 신호 사고”로 표시하는 안전장치다.

문제

손목 값이 튄다

최신 run에서는 valid-to-valid raw wrist jump가 여러 번 나왔고, 결국 TRACKING_LOSS로 실패했다.

수집 전

더 오래 안정 확인

첫 valid 3 frame만 보고 바로 시작하지 않고, 기본 15 frame 동안 right-wrist가 안정적인지 본다.

저장 후

학습 후보에서 분리

튀는 값이 저장되더라도 raw evidence는 보존하고, RAW_WRIST_JUMP 탈락 이유를 남긴다.

Implementation

이번 step에서 바꾼 3개 안전문

1. Evaluator gate

RAW_WRIST_JUMP 추가

action.raw_wrist_direct.valid_to_valid_jump_m가 기본 0.10m를 넘으면 task 성공 여부와 별개로 DATA_QUALITY_FAILURE가 된다.

2. Recenter preflight

첫 recenter 전에 stable right-wrist window 확인

first_valid_hand recenter는 이제 valid frame 개수뿐 아니라 연속 wrist jump가 RDF_AUTO_RECENTER_STABLE_M 이하인지 본다.

3. Debug default

run_hmd_axis_debug.sh 기본값 강화

HMD axis debug는 기본 RDF_WARMUP_VALID_FRAMES=15, RDF_AUTO_RECENTER_VALID_FRAMES=15, RDF_AUTO_RECENTER_STABLE_M=0.03로 시작한다.

Evidence

테스트로 확인한 것

먼저 실패하는 테스트를 만들고, 그 테스트가 통과할 만큼만 구현했다. 실제 HMD 실증은 다음 step으로 남겼다.

RED

구현 전 실패

RAW_WRIST_JUMP gate, stable recenter source marker, debug script 기본값 테스트가 모두 실패했다.

GREEN

타깃 테스트 통과

세 테스트가 통과하여 evaluator와 preflight 기본 방어선이 코드상 반영되었음을 확인했다.

STOP

HMD 실증 전 정지

이번 step은 실제 장비 테스트를 실행하지 않는다. 다음 실증 run은 사람이 HMD를 착용하고 별도로 수행한다.

uv run pytest \
  apps/api/tests/test_evaluator.py::test_evaluator_raw_wrist_valid_to_valid_jump_gate \
  apps/api/tests/test_teleop_diagnostics_scripts.py::test_live_teleop_auto_recenter_requires_stable_right_wrist_window \

2026-05-31 update

이제 들어갈 실증은 Gate A 수집이 아니라 Gate 0 입력 검증이다

방금 작업의 결론은 “바로 collection을 재개한다”가 아니다. 먼저 Quest/OpenXR 입력 스트림이 robot action label로 쓸 만큼 안정적인지 gate0-all로 증명해야 한다. 통과 전에는 run_collection_loop.sh가 시작 전에 막힌다.

추가된 경계

공통 입력 샘플

WristPoseSample과 InputSourceAdapter를 추가해 Quest/OpenXR handtracking을 첫 adapter로 분리했다.

Fail closed

출처 없는 입력은 통과 금지

UNKNOWN_INPUT_SOURCE와 UNSUPPORTED_INPUT_SOURCE는 Gate 0 fail이다. 미래 source가 Quest처럼 몰래 pass하지 않는다.

Hard block

Gate A collection 잠금

최신 fresh gate0_all aggregate가 네 stage와 input source 검증을 모두 통과해야만 collection loop가 시작된다.

1. Run

물리 HMD Gate 0 batch 실행

cd ~/robot-data-forge
./scripts/run_hmd_axis_debug.sh gate0-all
2. Read

최신 aggregate 확인

ls -t storage/logs/hmd_axis_debug/*.gate0_all.json | head -1

gate0_all_pass=true와 gate_a_collection_allowed=true가 둘 다 필요하다.

3. Decide

PASS일 때만 Gate A로 이동

RDF_COLLECTION_LOOP_GATE0_PREFLIGHT_ONLY=1 ./scripts/run_collection_loop.sh
./scripts/run_collection_loop.sh
  • RAW_WRIST_JUMP, TRACKING_LOSS, AUTO_RECENTER_UNSTABLE_RIGHT_WRIST는 task 실패가 아니라 input/control quality 실패다.
  • UNKNOWN_INPUT_SOURCE 또는 UNSUPPORTED_INPUT_SOURCE가 나오면 adapter/source contract를 먼저 고친다.
  • Gate 0이 false이면 axis/gain 튜닝이나 collection 재개로 넘어가지 않는다.

How to read the next HMD run

다음 HMD 실증에서 볼 신호

  • AUTO_RECENTER_UNSTABLE_RIGHT_WRIST가 반복되면 아직 수집하면 안 된다.
  • failure_reason=RAW_WRIST_JUMP가 나오면 task 실패가 아니라 입력 품질 실패로 본다.
  • failure_reason=TRACKING_LOSS가 계속 나오면 axis/gain 튜닝을 미루고 handtracking 환경을 먼저 손본다.
  • right_hand_tracked_rate, xr_frame_valid_rate, H13이 좋아진 뒤에만 Gate A collection을 재개한다.
중요: 이번 문서는 “실증 테스트 준비 완료”가 아니라 “실증 테스트 전에 추가해야 할 방어선 1차 완료”를 뜻한다. 실제 HMD run을 통과하기 전에는 accepted trajectory collection을 재개하지 않는다.
Claim boundary: RDF는 raw trajectory를 버리지 않는다. 대신 task success, data quality, replay/action contract를 분리해 학습 가능한 dataset artifact만 선별한다. 실제 HMD 실증은 이 step 이후에 별도로 수행한다.
